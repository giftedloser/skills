# Repository Presentation Quality Bar

Use this as a judgment aid, not a completeness checklist. Apply only the parts the repository and its audience need.

## Identity

- A first-time reader can name the project, its purpose, and its intended user after the opening screen.
- The voice and visual treatment feel specific to this project rather than transferable to any software repository.
- Existing distinctive language and assets are preserved unless they are inaccurate or obstruct comprehension.
- The title, description, screenshots, package identity, and repository metadata do not contradict one another.

## Hierarchy

- The first viewport gives the reader a useful next action or concrete proof, not a wall of badges or decoration.
- Sections follow reader questions in their likely order; they do not follow a generic README template.
- Secondary detail is linked or collapsed only when doing so improves navigation.
- Tables and diagrams replace genuine complexity; they do not restate prose.

## Copy

- Claims are concrete, evidence-backed, and written in the project's natural vocabulary.
- Commands, paths, feature names, and compatibility statements match repository evidence.
- Empty adjectives and generic launch copy are removed.
- Limitations and maturity are stated when they affect a reader's decision.
- Short is acceptable. Missing conventional sections are acceptable when the subject does not need them.

## Visuals

- Every visual has a job: identity, demonstration, comparison, or explanation.
- New art derives from actual project concepts or supplied brand material.
- Screenshots show the current product and crop toward the important interaction.
- Assets remain legible in GitHub light and dark themes and at narrow widths.
- Alt text describes useful content; decorative art uses empty alt text where appropriate.
- Avoid generic gradients, glowing shapes, stock device frames, fake terminals, logo clouds, emoji headings, and motion without meaning.

## Trust

- Links, images, badges, and examples resolve.
- Badges reflect live sources and answer a reader question.
- No claim implies security, performance, adoption, support, release status, or production readiness without proof.
- No content is added merely because another polished repository has it.
- Repository settings and public metadata remain proposals until the user approves them.

## Stop conditions

Stop when the main presentation problem is fixed. Do not add a logo because the README changed, a diagram because a logo exists, or templates because the repository now looks maintained. A clean no-change recommendation is a successful outcome.
